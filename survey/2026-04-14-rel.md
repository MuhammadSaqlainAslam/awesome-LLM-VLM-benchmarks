# REL: Evaluating Relational Reasoning in LLMs (2026)

## Problem
Existing benchmarks conflate relational reasoning difficulty with the sheer number of entities, masking a more fundamental bottleneck: the arity of relations (how many variables must be bound simultaneously). There is no benchmark that isolates the minimum number of simultaneously co-bound variables — "Relational Complexity" — while keeping total entity count fixed, making it impossible to diagnose whether LLM failures stem from combinatorial overload or from a genuine incapacity to handle higher-arity relational constraints.

## Method
**REL** (arXiv: 2604.12176, April 14, 2026) introduces the concept of **Relational Complexity (RC)** — the minimum number of independent elements that must be simultaneously bound to satisfy a relation — and constructs a generative benchmark framework that systematically varies RC across three scientific domains: **algebra**, **chemistry**, and **biology**. Problems are procedurally generated to hold total entity count constant while varying only RC, enabling clean ablation. The benchmark spans from binary relations (RC=2) through higher-arity relations, isolating the arity bottleneck independently of other confounding factors such as domain knowledge depth or surface lexical difficulty.

Authors: Lukas Fesser, Yasha Ektefaie, Ada Fang, Sham M. Kakade, Marinka Zitnik.

## Benchmarks / Datasets
- **REL** generative benchmark: algebra, chemistry, biology domains
- Systematic variation of Relational Complexity (RC) with fixed total entity counts
- Spans binary (RC=2) to higher-arity relations
- **Metrics:** Accuracy by RC level; performance degradation slope across RC levels

## Key Results

| Finding | Detail |
|---|---|
| Performance vs. RC | Consistent decline as RC increases, even with fixed entity count |
| Compute scaling | Degradation persists despite increased inference-time compute |
| Few-shot learning | Few-shot examples do not recover performance at high RC |
| Root cause | Bottleneck is higher-arity binding, not insufficient reasoning steps |

- **Model performance degrades consistently as Relational Complexity increases**, even when total entity count is held fixed — ruling out entity count as the confounding factor
- The performance ceiling persists under increased computational resources (more tokens, chain-of-thought), indicating a structural limitation in handling higher-arity relational constraints
- Few-shot prompting fails to bridge the RC gap, suggesting that neither in-context learning nor extended reasoning resolves the arity bottleneck
- Traditional benchmarks that vary entity count without controlling for arity likely underestimate the depth of LLM relational reasoning failures

## FoxBrain Relevance
REL pinpoints a previously unmeasured LLM blind spot directly relevant to FoxBrain's enterprise use cases: supply chain dependency reasoning, multi-party contract constraint checking, and multi-variable manufacturing optimization all require high-arity relational binding. The RC metric is immediately applicable as a diagnostic: FoxBrain tasks involving more than two simultaneously co-bound variables (e.g., supplier × component × spec × lead-time × cost constraints) should be stress-tested against the RC degradation curve before deployment. The benchmark's open generative framework can be adapted to manufacture FoxBrain-specific relational test cases.
